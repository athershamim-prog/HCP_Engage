# Phase 2 — UI Design Contract

> Visual and interaction contract for Phase 2: FMV + Engagement.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
> Requirements: FMV-01, FMV-02, FMV-03, FMV-04, FMV-05, ENG-01, ENG-02, ENG-03.
>
> This contract EXTENDS Phase 1 (01-UI-SPEC.md). All spacing, typography, color, and
> interaction patterns from Phase 1 carry forward unchanged. Only additions and Phase 2-
> specific screens are declared here.

---

## Design System

| Property | Value | Source |
|----------|-------|--------|
| Tool | shadcn/ui | Carry forward — Phase 1 |
| Preset | Default neutral (initialized in Phase 1) | Phase 1 UI-SPEC |
| Component library | Radix UI (via shadcn) | Carry forward — Phase 1 |

neutral. FMV and engagement screens carry the same register — no decorative elements, no
color-coded urgency beyond the badge color maps defined below.

## Spacing Scale

Identical to Phase 1 — carry forward without modification.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps, badge internal padding |
| sm | 8px | Compact row padding, input icon insets |
| md | 16px | Default card padding, form field gaps |
| lg | 24px | Section headings to content, sidebar item groups |
| xl | 32px | Card-to-card gaps, main content padding |
| 2xl | 48px | Page section breaks |
| 3xl | 64px | Top-of-page hero padding (login page only) |

Exceptions (Phase 1 exceptions carry forward):
- Touch target minimum: 44px height for all interactive controls (WCAG 2.5.5)
- Table row height: 48px (compact) — same as HCP list
- Status badge height: 24px (fixed)

Phase 2 additions:
- FMV rate reference panel minimum height: 80px — prevents layout shift when rate loads after HCP + type selection
- Upload drop zone height: 128px

## Typography

Identical to Phase 1 — carry forward without modification.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 14px | 400 (Regular) | 1.5 | Table cells, form labels, detail text |
| Label | 12px | 600 (Semibold) | 1.4 | Column headers, badge text, helper text, metadata |
| Heading | 20px | 600 (Semibold) | 1.2 | Page titles, card titles |
| Display | 28px | 600 (Semibold) | 1.15 | Login page heading only (Phase 1) |

**Font family:** `Inter, ui-sans-serif, system-ui, -apple-system, sans-serif`

**Weights in use:** 400 (Regular) and 600 (Semibold) only. No other weights are permitted.

**Truncation rule:** All table cell text truncates at one line with an ellipsis tooltip on hover.
HCP name, NPI, and engagement type are never truncated.

Phase 2 additions:
- FMV rate display value (e.g., "$350.00/hour"): Body 14px / 400 — not given special large treatment;
  it is reference data, not a highlighted number
- Rate card version label (e.g., "v3 — Active"): Label 12px / 600

## Color

### Base palette (carry forward from Phase 1)

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `hsl(0 0% 98%)` — near-white | Page background, table alternating rows |
| Secondary (30%) | `hsl(220 13% 18%)` — slate-900 | App shell sidebar, top nav, header bar |
| Accent (10%) | `hsl(221 83% 53%)` — blue-600 | See reserved list below |
| Destructive | `hsl(0 72% 51%)` — red-600 | Destructive actions, rejection states |
| Warning | `hsl(38 92% 50%)` — amber-500 | Pending rate card badge, caution states |
| Success | `hsl(142 71% 45%)` — green-600 | Active rate card badge, approved states |
| Muted | `hsl(215 16% 47%)` — slate-500 | Secondary text, helper text, disabled states |

**Accent (`hsl(221 83% 53%)`) reserved for (Phase 1 list + Phase 2 additions):**
1. Primary action buttons: "Search NPI", "Add HCP", "Run Debarment Check", "Save Determination",
   "Set Status" (Phase 1)
2. Phase 2 primary action buttons: "Upload Rate Card", "Activate Rate Card", "Submit for Approval",
   "Save Draft", "Approve"
3. Active navigation item highlight in sidebar
4. Focused form input ring
5. Pagination active page indicator
6. Submitted engagement status badge background (see badge map below — submitted maps to blue-600
   which is the accent color)

**Accent is NOT used for:** headings, table row highlights, informational callouts, decorative
elements, secondary buttons, the FMV rate reference panel background.

### Screen 3: FMV Rate Card Upload Flow (Wizard)

**Route:** `/fmv/upload` (Step 1) → same route with server state (Step 2 preview)
**Accessed by:** Compliance only
**Layout:** Centered single-column form (720px max-width) inside app shell; wizard progress
indicator at top

**Wizard step indicator:**
- Two steps displayed as horizontal step bar: "1 Upload File" → "2 Review & Activate"
- Active step: accent underline + semibold label
- Completed step: muted label + Lucide `Check` icon
- Step bar height: 48px

**Step 1: Upload File**

Components:
- Page heading: "Upload FMV Rate Card" (Heading / 20px / semibold)
- Drop zone: dashed border (`border-dashed border-2 border-slate-300`), 128px height, centered
  content:
  - Lucide `FileSpreadsheet` icon (24px, muted color)
  - Label: "Drag and drop your Excel or CSV file here" (Body / 14px)
  - Subtext: "Or click to browse — .xlsx, .xls, .csv accepted" (Label / 12px / muted)
- File selected state: drop zone collapses to a single row showing file name + size + Lucide `X`
  button to remove
- "Upload and Parse" accent `Button` — disabled until a file is selected
- "Cancel" ghost `Button` — navigates back to `/fmv`

Required file format hint (below drop zone):
- shadcn `Card` with Lucide `Info` icon + heading "Expected column format" (Label / 12px / semibold):
  - Monospace code block listing: `specialty_code | state | engagement_type | rate_usd | rate_unit`
  - Note: "state: 2-letter abbreviation or blank for national rate"
  - Note: "rate_unit: per_hour | per_day | per_event | flat_fee"

Interaction contract:
- Accept: `.xlsx`, `.xls`, `.csv` — browser `accept` attribute enforced; wrong file type shows
  inline error "Only .xlsx, .xls, and .csv files are accepted."
- Max file size: 5 MB — enforced client-side before upload; error "File exceeds the 5 MB limit.
  Split the file and re-upload."
- "Upload and Parse" triggers Server Action; during parse: button shows spinner + "Parsing..."
  text, disabled; drop zone unchanged
- Parse success → advances to Step 2 with parsed data in server state (no client-side storage)
- Parse failure (malformed file / unreadable): inline error below drop zone —
  "Could not parse this file. Verify the format and try again." — Step 1 remains active

**Step 2: Review and Activate**

Components:
- Page heading: "Review Rate Card" (Heading / 20px / semibold)
- Summary bar (shadcn `Card`, compact): shows parsed row count, valid row count, invalid row count
  - If invalid rows exist: amber inline alert "X rows have unrecognized specialty codes and will
    prevent activation. Fix the file and re-upload, or abort."
  - If all rows valid: green inline alert "All {N} rows passed NUCC validation."
- Preview table — shadcn `Table` — all parsed rows:

| Column | Notes |
|--------|-------|
| # | Row number |
| Specialty Code | Raw value from file |
| Specialty Name | Resolved NUCC display name (blank if unrecognized) |
| State | 2-letter abbreviation or "National" if blank |
| Engagement Type | Raw value from file |
| Rate | `$XXX.XX` formatted |
| Rate Unit | Display: "per hour" / "per day" / "per event" / "flat fee" |
| Status | Validation badge: `valid` (green) or `unrecognized` (red) |

- Table pagination: 50 rows per page if total > 50; otherwise no pagination
- "Activate Rate Card" accent `Button` — disabled if ANY row has `unrecognized` status
- "Re-upload File" ghost `Button` — navigates back to Step 1; clears parsed state
- "Cancel" ghost `Button` — navigates to `/fmv`; clears parsed state

Interaction contract:
- "Activate Rate Card" button: disabled + tooltip "Fix unrecognized rows before activating" when
  invalid rows exist
- On activate: Server Action sets `effectiveFrom = now()`, marks prior active card `effectiveTo =
  now()`, persists all valid rows as `FmvRate` records
- Activation success: redirect to `/fmv` with success toast "Rate card activated. Prior version
  superseded."
- Activation failure (server error): inline error above button "Activation failed. Try again or
  contact your administrator." — stays on Step 2
- No back-navigation to Step 1 once Step 2 is rendered unless "Re-upload File" is clicked

### Screen 6: New Engagement Form

**Route:** `/engagements/new`
**Accessed by:** Business, Compliance
**Layout:** Two-column layout — left 65% form, right 35% FMV reference panel

**Left column: Engagement form**

- Page heading: "New Engagement" (Heading / 20px / semibold)
- Subheading: "Fill in the details below. Save as a draft to continue later, or submit directly for
  approval." (Body / muted)

Form fields (all required unless marked optional):

| Field | Component | Notes |
|-------|-----------|-------|
| HCP | `Input` with search + popover suggestion list | Search by name or NPI; displays "Full Name — NPI" in dropdown |
| Engagement Type | `Select` | 5 options using display labels from engagement type map |
| Proposed Date | `Input` type="date" | Future date only enforced client-side |
| Compensation (USD) | `Input` type="number" min="0" step="0.01" | Prepend `$` symbol inside input |
| Description / Scope of Work | `Textarea` rows=4 | Required; min 20 chars |

- "Save Draft" ghost `Button` (secondary, border + text, no accent fill) — saves without
  submitting; navigates to engagement detail on success
- "Submit for Approval" accent `Button` — saves and submits (Draft → Submitted); navigates to
  engagement detail on success
- "Cancel" ghost `Button` — confirms with "Discard unsaved engagement?" if any field has been
  touched; navigates to `/engagements` on confirm

**Right column: FMV Rate Reference Panel**

- Panel appears as a shadcn `Card` in the right sidebar
- Initial state (HCP or engagement type not yet selected):
  - Heading: "FMV Rate Reference" (Heading / 20px / semibold)
  - Body: "Select an HCP and engagement type to see the applicable FMV rate." (Body / muted)
  - Panel background: `hsl(0 0% 98%)` (same as page — no special background)

- Loaded state (both HCP and engagement type selected, rate found):
  - Heading: "FMV Rate Reference" (Heading / 20px / semibold)
  - Rate line: "{NUCC specialty display name} / {engagement type display label} / {state or
    National}" (Label / 12px / muted)
  - Rate value: "$XXX.XX per hour" (or appropriate unit) (Body / 14px / 400)
  - Disclaimer: "Shown for reference only. Submission is not blocked." (Label / 12px / muted /
    italic)

- No-rate state (both selected, no matching rate row in active rate card):
  - Same heading
  - Body: "No FMV rate on file for {NUCC specialty display name} / {engagement type display
    label} / {state}." (Body / muted)
  - Disclaimer: "Submission is not blocked — document the basis for compensation in the scope
    field." (Label / 12px / muted / italic)

- No active rate card state:
  - Body: "No active FMV rate card. Contact your compliance officer." (Body / muted)

**Interaction contract:**
- HCP field: shows popover suggestion list after 2 characters typed; list shows up to 8 results;
  each result: "Full Name — NPI (NUCC specialty display name)"; clicking selects HCP and clears
  input replacing it with the selected name + an `X` to deselect
- When HCP selected: FMV panel initiates rate lookup using HCP's `nuccCode` and `primaryState`;
  shows skeleton loader (two lines, 100% and 60% width) during fetch
- When engagement type changes: FMV panel re-fetches with new type; shows skeleton loader again
- "Save Draft" validation: HCP required + at least one other field touched; if HCP missing, inline
  error "Select an HCP before saving."
- "Submit for Approval" validation: all fields required; inline errors on blur; button disabled
  until all fields valid and minimum lengths met
- Description field: character counter shown below ("47/20 minimum characters")
- Compensation field: no minimum enforced in v1 (FMV blocking is v2); accepts $0.00 with no warning
- Proposed date: past dates show inline warning "This date is in the past. Confirm this is
  correct." — does not block submission
- Both action buttons show loading spinner while Server Action is in flight; both disabled during
  loading

   - Rendered inline below the Status History card, outside the card, as an amber-bordered

The right panel renders differently per role and status combination. In all cases it is a
shadcn `Card`.

| Status | Role | Panel content |
|--------|------|---------------|
| draft | Owner (Business/Compliance) | "Edit" ghost button + "Submit for Approval" accent button + "Delete Draft" destructive link |
| draft | Non-owner | Read-only panel: "Draft — waiting for submission" (muted) |

| submitted | Business | "Pending Approval" status display (no actions) |

| rejected | Any | "Rejection Reason" display (duplicate of left column callout; no actions) |

**Approve/Reject form (for Compliance and Finance on submitted engagements):**
- Card heading: "Review Engagement" (Heading / 20px)
- "Approve" accent `Button` (full width in card)
- Horizontal divider with "or" text
- Rejection section:
  - `Textarea` label: "Rejection Reason" with red asterisk (required)
  - `Textarea` placeholder: "Explain why this engagement is being rejected..."
  - Minimum 10 chars enforced client-side
  - "Reject" destructive `Button` (full width) — enabled only when reason field meets minimum
  - Character counter below textarea

**"Delete Draft" destructive link:**
- Rendered as a small destructive-colored text link below the two draft action buttons
- Copy: "Delete this draft"
- Click triggers a shadcn `AlertDialog` confirmation modal (the ONLY use of a modal in Phase 2 —
  justified because deletion is irreversible):
  - Heading: "Delete engagement draft?"
  - Body: "This action cannot be undone. The draft will be permanently deleted."
  - Confirm button: "Delete Draft" (destructive fill)
  - Cancel button: "Cancel" (ghost)

**Interaction contract:**
- "Submit for Approval" from the detail page: same validation as the New Engagement form submit;
  on success navigates back to same page with updated status badge
- "Approve": Server Action; on success page reloads with status = approved and success toast
  "Engagement approved."
- "Reject": Server Action; on success page reloads with status = rejected; rejection reason
  displays in left column callout; success toast "Engagement rejected."
- "Mark as Completed": Server Action; on success page reloads with status = completed; success
  toast "Engagement marked as completed."
- "Delete Draft": on confirm, Server Action deletes record; on success navigates to `/engagements`
  with success toast "Draft deleted."
- Business user accessing an engagement they do not own: 404 (server-side guard)

- Row click: navigates to `/engagements/[id]` for that engagement
- No pagination initially — queue is expected to be short in v1; add pagination if > 25 rows

- Empty state: see Copywriting Contract

  no inline approve/reject in the queue table itself

- No filtering beyond the implicit `status = submitted` filter — queue shows all submitted
  regardless of submitter

## Interaction Patterns

### Carry forward from Phase 1 (all patterns apply unchanged)

| Pattern | Implementation |
|---------|---------------|
| Loading states | Spinner inside triggering button + button disabled; no full-page skeleton for in-place operations |
| Toast notifications | shadcn `Sonner` — top-right, 4-second auto-dismiss, success/error variants |
| Form validation | Inline validation on blur (not on every keystroke); error message directly below the offending field |
| Mandatory field indicator | Red asterisk (*) in Label weight beside the field label |
| Role-based element hiding | Non-permitted elements are not rendered — complete absence |
| Table row navigation | Full row is clickable; cursor: pointer |
| Badge component | shadcn `Badge` — solid fill for status badges; `variant="outline"` for informational states |

### Phase 2 additions

| Pattern | Implementation |
|---------|---------------|
| Wizard progress indicator | Horizontal step bar; 2 steps; active step has accent underline + semibold label; completed step has muted label + Lucide `Check` icon |
| File drop zone | Dashed border, 128px height; drag-over state: `border-accent bg-blue-50` (subtle blue tint); file selected state collapses to filename row |
| Skeleton loader (FMV panel) | Two skeleton lines (100% width, 60% width) using shadcn Skeleton during rate fetch; renders inside the FMV reference panel |
| HCP search popover | Popover list of up to 8 matches; renders on 2+ characters typed; each item: "Full Name — NPI (Specialty)"; keyboard-navigable with Up/Down arrows + Enter to select |
| Activation confirmation | No modal — clicking "Activate Rate Card" (Step 2) immediately activates via Server Action with loading spinner in the button; the preview table serves as the confirmation step |
| Approve confirmation | No modal — "Approve" directly triggers Server Action with loading spinner |
| Reject confirmation | No modal — the mandatory rejection reason field IS the confirmation friction; "Reject" button only enables when reason is valid |
| Delete draft confirmation | AlertDialog modal — the ONLY modal in Phase 2; justified because deletion is irreversible and there is no recovery path |
| Table — no inline editing | All edits and actions occur on the detail page; engagement list and queue are read-only tables |
| URL-param filtering | Engagement list filters are URL searchParams; filter changes trigger server re-render (no client-side filtering); URL is shareable |
| Past date warning | Non-blocking inline warning below date field in amber text; does not disable submit |

## Accessibility Contract

### Carry forward from Phase 1

| Requirement | Implementation |
|-------------|---------------|
| Color contrast | All text on colored badges must meet WCAG AA (4.5:1) — white text on all solid-fill badges |
| Keyboard navigation | All interactive elements reachable by Tab; modals/panels trap focus |
| Focus ring | Accent color (`hsl(221 83% 53%)`) 2px offset ring on all focusable elements |
| Screen reader | `aria-label` on icon-only buttons; `role="status"` on badge elements; `aria-live="polite"` on in-place update areas |
| Form labels | Every input has a visible `<label>` — no placeholder-only labels |
| Touch targets | Minimum 44px height on all interactive controls |

### Phase 2 additions

| Requirement | Implementation |
|-------------|---------------|
| File drop zone | `role="button"` + `tabIndex=0` + `aria-label="Upload rate card file"` on drop zone div; keyboard Enter/Space triggers file browser |
| HCP search popover | `role="combobox"` on input, `role="listbox"` on popover, `role="option"` on each result; `aria-expanded` toggled; `aria-activedescendant` tracks keyboard selection |
| Wizard steps | `aria-current="step"` on active step; completed steps have `aria-label="{Step name} — completed"` |
| FMV panel live region | `aria-live="polite"` on the FMV rate reference panel — announces rate value when it loads after HCP/type selection |
| AlertDialog (delete draft) | shadcn `AlertDialog` uses Radix `AlertDialog.Root` — focus is trapped; `aria-describedby` wired to modal body text; Escape key dismisses |
| Status history timeline | Rendered as `<ol>` with `<li>` items — semantic list structure for screen reader enumeration |
| Approval queue count chip | `aria-label="{N} engagements pending approval"` |
