# Phase 1 — UI Design Contract

> Visual and interaction contract for Phase 1: Auth + HCP Management.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
> Requirements: AUTH-01, HCP-01, HCP-02, HCP-03, HCP-04.

---

## Design System

| Property | Value | Source |
|----------|-------|--------|
| Tool | shadcn/ui | CLAUDE.md stack definition |
| Preset | Initialize with `npx shadcn@latest init` during Phase 1 setup; use default neutral style | default |
| Component library | Radix UI (via shadcn) | CLAUDE.md stack definition |
| Icon library | Lucide React (shadcn default) | default |
| Font | Inter (system-ui fallback chain) | default — enterprise legibility |
| CSS approach | Tailwind CSS v4 utility classes; shadcn CSS variables for theming | CLAUDE.md stack definition |

**Design intent:** Enterprise compliance tool. Tone is professional, audit-ready, and neutral. No playful elements, no decorative gradients. Every screen must inspire trust in a regulated environment.

---

## Spacing Scale

Declared values (multiples of 4 only):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps, badge internal padding |
| sm | 8px | Compact row padding, input icon insets |
| md | 16px | Default card padding, form field gaps |
| lg | 24px | Section headings to content, sidebar item groups |
| xl | 32px | Card-to-card gaps, main content padding |
| 2xl | 48px | Page section breaks |
| 3xl | 64px | Top-of-page hero padding (login page only) |

Exceptions:
- Touch target minimum: 44px height for all interactive controls (WCAG 2.5.5)
- Table row height: 48px (compact) for HCP list — denser than standard 56px to fit audit tables
- Status badge height: 24px (fixed, not subject to line-height scaling)

## Typography

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 14px | 400 (Regular) | 1.5 | Table cells, form labels, detail text |
| Label | 12px | 600 (Semibold) | 1.4 | Column headers, badge text, helper text, metadata |
| Heading | 20px | 600 (Semibold) | 1.2 | Page titles, card titles |
| Display | 28px | 600 (Semibold) | 1.15 | Login page heading only |

**Font family:** `Inter, ui-sans-serif, system-ui, -apple-system, sans-serif`

**Weights in use:** 400 (Regular) and 600 (Semibold) only. No other weights are permitted.

**Truncation rule:** All table cell text truncates at one line with an ellipsis tooltip on hover. HCP name and NPI are never truncated.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `hsl(0 0% 98%)` — near-white | Page background, table row alternating (odd rows: white, even rows: `hsl(0 0% 96%)`) |
| Secondary (30%) | `hsl(220 13% 18%)` — slate-900 | App shell sidebar, top nav, header bar |
| Accent (10%) | `hsl(221 83% 53%)` — blue-600 | Reserved for specific elements only — see below |
| Destructive | `hsl(0 72% 51%)` — red-600 | "Do Not Engage" status badge, destructive action buttons, debarment hit indicators |
| Warning | `hsl(38 92% 50%)` — amber-500 | "Debarment check not run" warning badge, suspended status badge |
| Success | `hsl(142 71% 45%)` — green-600 | "Active" status badge, debarment no-hit indicator, check-passed states |
| Muted | `hsl(215 16% 47%)` — slate-500 | Secondary text, helper text, disabled states |

**Accent (`hsl(221 83% 53%)`) reserved for:**
1. Primary action buttons only: "Search NPI", "Add HCP", "Run Debarment Check", "Save Determination", "Set Status"
2. Active navigation item highlight in sidebar
3. Focused form input ring
4. Pagination active page indicator

**Accent is NOT used for:** headings, table row highlights, informational badges, decorative elements, secondary buttons (secondary buttons use border + text, no fill).

**Status badge color map (required — used in HCP profile and HCP list):**

| HCP Status | Badge Color |
|------------|-------------|
| active | green-600 background / white text |
| inactive | slate-400 background / white text |
| suspended | amber-500 background / white text |
| do-not-engage | red-600 background / white text |

**Debarment result badge color map:**

| Result | Badge Color |
|--------|-------------|
| No hit | green-600 / white |
| Match found | red-600 / white |
| Not checked | slate-400 / white |
| Cleared (determination) | green-600 outline / green-700 text |
| Confirmed exclusion | red-600 outline / red-700 text |
| False positive | amber-500 outline / amber-700 text |

## Screen Inventory

Seven screens are in scope for Phase 1. Each has a declared layout, key components, and interaction contracts.

---

### Screen 1: Login Page

**Route:** `/sign-in` (Clerk hosted page or `<SignIn />` embedded)
**Accessed by:** All users
**Layout:** Centered card on full-bleed neutral background

**Components:**
- shadcn `Card` (480px max-width, center-aligned)
- Clerk `<SignIn />` embedded component OR redirect to Clerk-hosted page
- Logo / product name above card: "HCP Engage" in Display size (28px / 600)
- Tagline below logo: "Pharma commercial compliance" in Label size (12px / 400 / muted color)

**Interaction contract:**
- No social login buttons rendered (email+password only per D-06)
- No "Sign up" link rendered (invited users only per D-07)
- Error on invalid credentials: Clerk-native inline error message — do not override Clerk error UI
- Redirect on success: role-determined (Business/Compliance → `/hcps`, Finance → `/dashboard` placeholder)

**Copy:**
- Page title (browser tab): "Sign In — HCP Engage"
- Card heading: "Welcome back"
- Card subheading: "Sign in to your compliance workspace"

---

### Screen 2: App Shell (Role-Gated Navigation)

**Route:** All authenticated routes
**Accessed by:** All authenticated users
**Layout:** Fixed left sidebar (240px) + top header bar (56px) + main content area

**Sidebar navigation items by role:**

| Nav Item | Route | Business | Compliance | Finance |
|----------|-------|----------|------------|---------|
| HCP Directory | `/hcps` | Yes | Yes | No |
| Add HCP | `/hcps/new` | Yes | Yes | No |
| Dashboard | `/dashboard` | No | No | Yes (placeholder) |

*Finance sees a minimal shell with a Dashboard placeholder in Phase 1 — their primary surface is Phase 2 (engagement approval).*

*Compliance users with role expansion (D-04b) see the union of their granted roles' nav items.*

**Sidebar components:**
- Product logo / name at top (fixed, not scrollable)
- Nav items: `Button` variant ghost + Lucide icon + label
- Active item: accent fill background, white text
- Role label at bottom of sidebar: e.g., "Compliance Officer" in Label size / muted color
- Department tag shown for Business users: e.g., "Marketing" in Label size / muted color beneath role label

**Top header bar:**
- Left: current page title (Heading size / semibold)
- Right: user avatar (Clerk `<UserButton />`) + full name in Body size

**Interaction contract:**
- Navigation items not permitted for user's role: do not render — not rendered as disabled, completely absent
- Finance user in Phase 1: sidebar renders Dashboard only; HCP items not present
- Role expansion (D-04b): middleware computes effective permissions; nav renders accordingly
- All protected routes: redirect unauthenticated users to `/sign-in`; redirect wrong-role users to their permitted root

---

### Screen 3: HCP Directory (List)

**Route:** `/hcps`
**Accessed by:** Business, Compliance
**Layout:** Full-width data table with filter bar above

**Components:**
- Page heading: "HCP Directory" (Heading / 20px / semibold)
- Search/filter bar: shadcn `Input` (search by name or NPI, 320px width) + `Select` for status filter
- "Add HCP" button: accent-filled `Button`, right-aligned in header row
- shadcn `Table` with columns:

| Column | Width | Notes |
|--------|-------|-------|
| HCP Name | 22% | Full name, bold, links to HCP profile |
| NPI | 10% | 10-digit number, monospace font |
| Credentials | 10% | e.g., "MD, PhD" |
| Specialty | 18% | NUCC display name (not raw code) |
| Primary State | 8% | 2-letter abbreviation |
| Status | 12% | Color badge (see color map) |
| Debarment | 12% | Color badge (see color map) |
| Last Updated | 8% | Relative time, e.g., "2 days ago" |

- Row click: navigates to HCP profile (`/hcps/[id]`)
- shadcn `Pagination` component below table

**Interaction contract:**
- Table: no inline editing — all edits on HCP profile page
- Status filter: multi-select; default = all statuses shown
- Search: debounced 300ms; filters name (contains) and NPI (exact prefix match)
- Empty state (no HCPs in system): see Copywriting Contract
- Empty state (search returned no results): see Copywriting Contract
- Debarment "Not checked" badge: renders amber warning icon (Lucide `AlertTriangle`) beside badge
- Row hover: `hsl(220 14% 96%)` background (subtle, not blue)

---

### Screen 4: HCP Search / NPI Lookup (Add HCP Flow)

**Route:** `/hcps/new`
**Accessed by:** Business, Compliance
**Layout:** Centered single-column form (640px max-width) inside app shell

**Components:**
- Page heading: "Add HCP" (Heading / 20px / semibold)
- Subheading: "Search by NPI to pull verified HCP data from NPPES" (Body / muted)
- NPI input: shadcn `Input` (type="text", inputmode="numeric", maxlength=10) + "Search NPI" accent `Button` inline
- NPPES result card: shadcn `Card` rendered below input after successful lookup — shows pulled fields:
  - Full legal name (Heading / 20px)
  - Credentials (Label / 12px / muted)
  - NUCC specialty code + display name
  - Primary state
  - HCO affiliation (or "No affiliation on record" in muted text)
- "Add to Directory" accent `Button` at bottom of result card
- "Search again" ghost `Button` beside it

**Interaction contract:**
- NPI input: numeric only, 10 digits; client-side length validation before API call
- Search: calls NPPES API; loading state = spinner inside "Search NPI" button, button disabled
- NPPES not found (valid format, no record): inline error below input — "No HCP found for NPI {npi}. Verify the number and try again."
- NPPES API error (network/timeout): inline error — "NPPES lookup failed. Check your connection and try again."
- NPI already in system: result card renders with banner "This HCP is already in your directory." — "View Profile" button replaces "Add to Directory"
- "Add to Directory": creates local HCP record; on success navigates to new HCP profile page with success toast

---

### Screen 5: HCP Profile Page

**Route:** `/hcps/[id]`
**Accessed by:** Business (read-only), Compliance (read + edit status + run debarment)
**Layout:** Two-column layout — left 65% main content, right 35% sidebar panel

**Left column sections (top to bottom):**

1. **Profile header:**
   - Full legal name (Display / 28px / semibold)
   - Credentials in muted Label (12px) beside name
   - NPI in monospace Label (12px) / muted
   - Current status badge (color-mapped)
   - Debarment badge

2. **NPPES Data section** (read-only, shadcn `Card`):
   - Section heading: "Verified HCP Data" (Heading / 20px)
   - Sub-label: "Source: NPPES — pulled {date}" (Label / 12px / muted)
   - Fields in two-column grid: NUCC Specialty, Primary State, HCO Affiliation, Credentials

3. **Debarment Check section** (shadcn `Card`):
   - Section heading: "Debarment Check" (Heading / 20px)
   - Last checked timestamp or "Never checked" in muted Label
   - Current result badge
   - If checked: OIG LEIE result row + SAM.gov result row (each with match badge)
   - If match found: match detail block (indented, border-left accent or destructive depending on source)
   - "Run Debarment Check" accent `Button` (Compliance only — hidden for Business)
   - Determination record (if recorded): rationale text + outcome badge + recorded-by + timestamp

4. **Status History section** (shadcn `Card`):
   - Section heading: "Status History" (Heading / 20px)
   - Timeline list: each entry shows status badge + reason text + set-by name + timestamp
   - Most recent entry at top
   - Empty: "No status changes recorded."

**Right sidebar panel:**

- **Set HCP Status** card (Compliance only):
  - `Select` for status (active / inactive / suspended / do-not-engage)
  - `Textarea` for reason (required, min 10 chars)
  - "Set Status" accent `Button`
  - Validation: reason is mandatory before save; button disabled until both fields filled
  - Hidden entirely for Business role

- **Quick Facts** card (both roles):
  - Date added to system
  - Added by (user name)
  - HCP record ID (internal, for support)

**Interaction contract:**
- Business role: right panel shows only "Quick Facts" card — "Set HCP Status" card is not rendered
- "Run Debarment Check" button: shows loading spinner; on completion refreshes debarment section in place (no full page reload)
- Debarment check results: OIG LEIE and SAM.gov results displayed in separate labeled rows; match = destructive badge + "View match details" expand toggle
- Match detail expand: inline accordion revealing raw match fields (name, exclusion type, exclusion date, NPI if available)
- Determination form: appears below match results only after check has been run and a match exists; always available to re-record even if previously recorded
- Status change: on save, status history section refreshes and the profile header badge updates; success toast confirms
- Reason field: character counter shown below textarea ("14/10 minimum characters")

---

### Screen 6: Run Debarment Check Flow

**Trigger:** "Run Debarment Check" button on HCP Profile (Screen 5)
**Accessed by:** Compliance only
**Pattern:** In-place panel update within HCP profile — no separate route

**States to render:**

| State | UI |
|-------|----|
| Idle (never checked) | "Run Debarment Check" button; "Never checked" muted label |
| Loading | Button becomes spinner + "Checking..." text, disabled; rest of section unchanged |
| No hit (OIG + SAM clear) | Green "No Hit" badge for each source; "Last checked {timestamp}" |
| Match found (one or both) | Red "Match Found" badge; match summary block with "Record Determination" section below |
| Error (check failed) | Amber inline alert: "Check failed — {reason}. Try again." |
| Determination recorded | Determination block: outcome badge + rationale + recorded-by + timestamp; "Update Determination" link to re-open form |

**Determination form fields (appears after match found):**
- `Select`: Outcome — "Cleared", "Confirmed Exclusion", "False Positive"
- `Textarea`: Rationale (required, min 20 chars)
- "Save Determination" accent `Button`
- "Discard Changes" ghost `Button` — collapses the determination form without saving

**Interaction contract:**
- Check can be re-run at any time regardless of prior determination
- Re-running check: prior determination is preserved visually but labeled "(Based on previous check — {date})" until a new determination is recorded
- Determination save: success toast "Determination recorded"; form collapses; determination block updates in place

---

### Screen 7: HCP Status Management (Panel on Profile)

**Trigger:** "Set HCP Status" panel on HCP Profile right sidebar (Screen 5)
**Accessed by:** Compliance only
**Pattern:** In-place form in right sidebar — no separate route

**Status option display in `Select`:**

| Option value | Display label |
|--------------|---------------|
| active | Active |
| inactive | Inactive |
| suspended | Suspended |
| do-not-engage | Do Not Engage |

**Interaction contract:**
- Selecting "do-not-engage": `Select` border changes to destructive color as a visual warning; reason field label changes to "Reason for Do-Not-Engage designation (required)"
- Reason field: required regardless of status choice; min 10 chars enforced client-side with inline validation message
- "Set Status" button: disabled until status selected + reason meets minimum length
- On save: profile header badge updates immediately (optimistic UI acceptable); status history section gains new entry at top
- Cannot set the same status that is already active: "Set Status" button disabled with tooltip "HCP is already {status}"

## Interaction Patterns (Cross-Screen)

| Pattern | Implementation |
|---------|---------------|
| Loading states | Spinner inside the triggering button + button disabled; no full-page skeleton for in-place operations |
| Toast notifications | shadcn `Sonner` (toast library) — top-right, 4-second auto-dismiss, success/error variants |
| Form validation | Inline validation on blur (not on every keystroke); error message directly below the offending field |
| Mandatory field indicator | Red asterisk (*) in Label weight beside the field label — not in the placeholder |
| Role-based element hiding | Non-permitted elements are not rendered (not disabled, not greyed out) — complete absence |
| Table row navigation | Full row is clickable; cursor: pointer; no separate "View" link column |
| Confirmation for destructive actions | No modal for status changes — mandatory reason field provides confirmation friction; no destructive deletions exist in Phase 1 |
| Badge component | shadcn `Badge` with `variant="outline"` for determination outcomes; solid fill for status and debarment hit/clear |

## Accessibility Contract

| Requirement | Implementation |
|-------------|---------------|
| Color contrast | All text on colored badges must meet WCAG AA (4.5:1 for normal text) — use white text on all solid-fill badges |
| Keyboard navigation | All interactive elements reachable by Tab; modals/panels trap focus |
| Focus ring | Accent color (`hsl(221 83% 53%)`) 2px offset ring on all focusable elements |
| Screen reader | `aria-label` on icon-only buttons; `role="status"` on badge elements; `aria-live="polite"` on in-place check result area |
| Form labels | Every input has a visible `<label>` — no placeholder-only labels |
| Touch targets | Minimum 44px height on all interactive controls |
